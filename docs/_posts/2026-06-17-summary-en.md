## [Stop Using JWTs: Controversial Argument vs. Community Insights](https://gist.github.com/samsch/0d1f3d3b4745d778f78b230cf6061452) ⭐️ 7.0/10

A developer's gist argues that JWTs are insecure for browser-based user sessions, but the community counters with proper usage patterns like service-to-service tokens, short lifetimes, refresh models, and DPoP (Demonstration of Proof-of-Possession). This debate brings clarity to JWT usage, separating anti-patterns from legitimate applications, and helps developers avoid security pitfalls while leveraging JWTs effectively. The original post highlights risks like token theft via XSS and lack of revocation, but ignores standards like DPoP (RFC 9449) which binds tokens to a sender's key, and manageable revocation lists since JWTs have expiry.

hackernews · dzonga · Jun 16, 16:49 · [Discussion](https://news.ycombinator.com/item?id=48558147)

**Background**: JSON Web Tokens (JWTs) are an open standard for securely transmitting information as a JSON object, often used for authentication. They can be signed to ensure integrity and optionally encrypted. Browser sessions often use JWTs stored in cookies or local storage, raising concerns about cross-site scripting (XSS) attacks. Refresh tokens help renew short-lived access tokens without re-authentication. DPoP is a mechanism to prove possession of a private key to prevent token replay.

**Discussion**: Commenters largely disagree with the absolutist claim, noting that JWTs are standard in OAuth2 and service-to-service auth. They recommend short-lived tokens, refresh mechanisms, and DPoP to address security concerns. Some argue the original post confuses implementation flaws with the standard itself.
